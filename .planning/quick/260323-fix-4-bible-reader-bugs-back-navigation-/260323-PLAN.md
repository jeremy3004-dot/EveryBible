<task type="auto">
  <name>Task 3: Fix audio follow-along verse advancing too early</name>
  <files>src/screens/bible/bibleReaderModel.ts</files>
  <action>
In `getEstimatedFollowAlongVerse` the current logic computes:

```typescript
const progressRatio = Math.max(0, Math.min(currentPosition / duration, 0.999999));
const weightedProgress = progressRatio * totalWeight;
```

And advances to a verse as soon as `weightedProgress` exceeds the verse's cumulative weight. This means the text advances to the NEXT verse the instant any progress enters that verse's weight range — i.e., it advances too early because there is no threshold/lag baked in.

Fix: Apply a small advance-delay factor so that the text switches to verse N only after a meaningful fraction of verse N's weight has elapsed. Concretely, change the verse match condition from `weightedProgress < cumulativeWeight` to `weightedProgress < cumulativeWeight - verseWeight * 0.25` (stay in the current verse until 25% into the next verse's window):

```typescript
let cumulativeWeight = 0;
for (const verse of verses) {
  const vw = getVerseWeight(verse);
  cumulativeWeight += vw;
  // Hold on the current verse until 25% of the next verse's window has elapsed,
  // preventing text from jumping ahead of audio playback.
  if (weightedProgress < cumulativeWeight - vw * 0.25) {
    return verse.verse;
  }
}
```

Wait — this logic inverts. Correct approach: stay on a verse until audio has passed the verse's START weight plus a threshold portion of its OWN weight. Rewrite the loop clearly:

```typescript
let cumulativeWeight = 0;
for (const verse of verses) {
  const vw = getVerseWeight(verse);
  const verseStart = cumulativeWeight;
  cumulativeWeight += vw;
  // Advance to this verse only after audio has passed 15% into its weight window.
  // This adds a small lag so text does not jump ahead of the audio.
  if (weightedProgress < verseStart + vw * 0.15) {
    // Still in the prior verse's territory (or just entered this one)
    // Return the PREVIOUS verse instead. But we can't easily do that here.
    // Use simpler approach: return this verse only when progress >= verseStart + threshold.
    // If not met, continue to see if progress fits a later verse — which it won't,
    // so we'll fall through to the last verse. Instead, break and return previous.
    break;
  }
  if (weightedProgress < cumulativeWeight) {
    return verse.verse;
  }
}
```

Actually the simplest safe fix that avoids logic errors: reduce the effective progress by a small lag offset before the comparison. Subtract a fixed fraction of total weight (e.g., 8%) from `weightedProgress` to delay all verse transitions uniformly:

```typescript
const progressRatio = Math.max(0, Math.min(currentPosition / duration, 0.999999));
// Apply a small lag (8% of total duration) so text does not advance ahead of audio.
const laggedProgress = Math.max(0, progressRatio - 0.08);
const weightedProgress = laggedProgress * totalWeight;
```

This is the simplest, least risky change: subtract an 8% lag from the progress ratio before computing which verse is active. The existing loop logic (`weightedProgress < cumulativeWeight`) is correct; only `progressRatio` needs adjustment.

Implement this approach (the lag offset) since it changes only 2 lines and keeps all existing loop logic intact.
