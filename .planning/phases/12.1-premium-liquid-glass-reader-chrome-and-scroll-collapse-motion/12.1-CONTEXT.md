# Phase 12.1: Premium liquid-glass reader chrome and scroll-collapse motion - Context

**Gathered:** 2026-03-21
**Status:** Ready for planning
**Source:** User-supplied reader video in `~/Downloads/ScreenRecording_03-21-2026 10-33-48_1.MP4`

<domain>
## Phase Boundary

Rebuild the read-mode Bible chapter experience so it matches the supplied video instead of the current static-header reader. The target is a premium, motion-led reading canvas: minimal glass controls floating above content at rest, then collapsing away as soon as the user starts scrolling.

Scope includes:

- replacing the static read-mode header with an overlay-based chrome system
- adding real blur-backed glass surfaces for the top controls and bottom floating controls
- switching read-mode typography toward an editorial serif treatment that better matches the reference
- implementing scroll-driven collapse so the top controls and supporting bottom controls shrink/fade into a small centered chapter pill
- removing the old bottom audio bar from read mode so the read surface matches the video's cleaner canvas

Out of scope:

- redesigning the dedicated listen-mode player around the earlier screenshot
- introducing new backend data sources or changing chapter-session routing
- replacing the wider app design system beyond reader-specific tokens and surfaces needed for this phase
  </domain>

<decisions>
## Implementation Decisions

### Locked Decisions

- The supplied video is the source of truth for the resting read-mode layout and the collapse motion.
- Read mode should open with top glass controls only: back, `Listen / Read`, and overflow.
- Read mode should not show the old static header title, translation chip, top `AA` button, top audio button, or bottom `AudioPlayerBar`.
- Bottom controls should begin as floating glass elements and collapse to a small centered chapter pill during scroll.
- The redesign must preserve current chapter-session continuity between `Listen` and `Read`.

### Claude's Discretion

- Exact scroll thresholds, easing, and opacity/scale timings for the collapse, as long as the result reads like the supplied motion reference.
- Whether the glass surfaces are implemented through one tiny helper component or screen-local composition.
- Exact mapping of the bottom-left floating control, as long as the existing translation/font flows remain accessible and the resulting layout stays faithful to the video.
- Whether reader-specific serif typography lives in the shared design system or locally inside the screen, as long as it remains maintainable.
  </decisions>

<specifics>
## Specific Ideas

- `BibleReaderScreen` already separates `listen` and `read`; this phase should deepen that split instead of trying to make one static header work for both.
- The current read screen uses a fixed header plus optional bottom audio bar. The video instead behaves like a full reading canvas with controls layered over it, so the screen should move to absolute overlays plus animated scroll progress.
- The contact-sheet review of the video shows three states:
  1. resting: top glass controls + serif chapter hero + bottom floating controls
  2. transition: chapter hero fades upward first while top controls remain briefly visible
  3. collapsed: top controls disappear and the bottom controls reduce to a centered chapter pill
- Expo already includes `expo-linear-gradient`, but not blur. A real liquid-glass approximation likely needs `expo-blur` plus translucent fills, borders, and shadow tuning.
  </specifics>

<deferred>
## Deferred Ideas

- Reworking listen mode to use the same bottom-control choreography if future feedback asks for parity there
- Full native screenshot-based visual regression for the reader
- A broader app-wide liquid-glass pass beyond the Bible reader
  </deferred>
